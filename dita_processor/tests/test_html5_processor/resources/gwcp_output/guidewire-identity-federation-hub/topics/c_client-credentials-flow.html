Using the Client Credentials OAuth 2.0 application flow

Connect application with trusted server-to-server communication, such as a "client" server that needs to access a "resource" server.

The client server has a copy of its client ID and client secret. It makes a single POST request to Guidewire Hub, specifying grant_type=client_credentials and an optional scope. The client ID and client secret are specified either by client_id and client_secret parameters, or by a Basic authorization header (for example, id:secret, Base64 encoded). Guidewire Hub returns an access token, and the client server uses this token to access the resource server.

Make system API calls using a token via the Client Credentials flow

The service application authenticates with Guidewire Hub to get a token. The application then uses this token to make REST calls against the ClaimCenter System APIs.

To get the token, the application issues the following call to Guidewire Hub as a POST:
authServerUri/v1/token?client_id=clientId&idp=idpId&response_type=token&
    scope=scope&client_secret=clientSecret&grant_type=client_credentials
authServerUri
The URI for Guidewire Hub.
clientId
A unique identifier for the application that the client uses in any requests to the Guidewire Hub.
idpId
The ID of your IdP that Guidewire provided you after onboarding your IdP.
scope
The scope chosen when registering the application, as well as cc.service and your tenant_id. Provide these as a space-separated list. For example, scope=cc.mycompany_allstarrepairintegration%20cc.service%20tenant.mycompany
clientSecret
The secret string shared between the client and Guidewire Hub.

This call instructs Guidewire Hub to verify the client secret. If the secret is verified, Guidewire Hub delivers a token back to the application. The application uses this token to make System API calls to ClaimCenter.
