Changes to implementation of gw.cloud.jobprocess.getUniqueOnPolicyPeriodBeansForValidation

The implementation of gw.cloud.jobprocess.getUniqueOnPolicyPeriodBeansForValidation has changed.

The implementation now uses a method on PolicyPeriod, PolicyPeriod#getUniqueOnPolicyPeriodBeans. The enhancement property getUniqueOnPolicyPeriodBeansForValidation has been removed, but you can preserve that implementation as content if you choose. When upgrading, you can do either of the following:
  • If you want to use the new method PolicyPeriod#getUniqueOnPolicyPeriodBeans, you must update gw.cloud.jobprocess.getUniqueOnPolicyPeriodBeansForValidation with a call to that method. This approach is preferred as it simplifies adding UniqueOnPolicyPeriod beans for each line of business. For example, LOB X adds its unique beans in the LOBXPolicyLineMethods implementation, LOB Y adds its unique beans in the LOBYPolicyLineMethods implementation, and so forth.
  • You can keep the existing implementation of gw.cloud.jobprocess.getUniqueOnPolicyPeriodBeansForValidation, but you must also keep the getUniqueOnPolicyPeriodBeansForValidation enhancement property called from that implementation. That enhancement method must add UniqueOnPolicyPeriod beans for all lines of business in your installation.

Changes to lines of business

Lines of business content has been moved from the pc-content module.

Business Owner’s Policy (BOP) content has been moved

Business Owner’s Policy (BOP) content has been moved from the pc-content module to the pc-lob-bop module.

If you have never used BOP to create a submission, there is no BOP policy data in the database. In this case, you can use a database upgrade trigger to remove BOP content from the product model and the data model. To remove BOP content, you must enable the trigger by adding the property service property dbupgrade.lob.BOPLine.cleanup and setting it to true. By default, the database upgrade is not triggered.

If you have used BOP to create a submission, you must enable BOP content by installing the BOP Guidewire Package Manager (GPM).

Commercial Auto (CA) content has been moved

Commercial Auto (CA) content has been moved from the pc-content module to the pc-lob-ca module.

If you have never used CA to create a submission, there is no CA policy data in the database. In this case, you can use a database upgrade trigger to remove CA content from the product model and the data model. To remove CA content, you must enable the trigger by adding the property service property dbupgrade.lob.BALine.cleanup and setting it to true. By default, the database upgrade is not triggered.

If you have used CA to create a submission, you must enable CA content by installing the BA Guidewire Package Manager (GPM).

The following configuration parameters have been removed.
Parameter Description

RateTableCacheUpdateIntervalSecs Obsolete and nonfunctional.

In a previous implementation of the rate table cache, all items in the cache were refreshed periodically. The interval between refresh events was specified using RateTableCacheUpdateIntervalSecs. In the current implementation, cache entries are refreshed based on the entry lifetime specified using RateTableCacheEntryLifetimeSecs. When a cache entry is accessed, if it has been in the cache longer than the configured lifetime, a refresh of that entry is triggered, thereby eliminating the need for periodically refreshing the entire cache.

ISPC‑3135 In previous releases, in non-production environments, changing a single rate table definition caused the rate table definition cache and the rate table cache to invalidate all cache entries, so these caches would be rebuilt from scratch. The rate table cache took a long time to rebuild if there were many rate tables or large rate tables. While the rate table cache was rebuilding, access to rate tables was slow.

In this release, when a rate table definition is changed, only the corresponding entry in the rate table definition cache is invalidated and, in the rate table cache, only those rate tables based on the changed definition are invalidated and rebuilt.

  • File upload is allowed only if the MIME type (media type) is defined in the mimetypemapping element in config.xml. If you need to upload a particular MIME type, you must add it to this configuration.

ISPL‑8103 Enhanced the REST framework to support the following new properties that you can add to JSON Schema properties, Swagger operations, and Swagger parameters: x-gw-deprecatedVersion, x-gw-deprecatedExtensionsVersion, x-gw-deprecatedReason.
  • All three properties appear when writing out JSON or Swagger schemas.
  • Adding one or both of x-gw-deprecatedVersion and x-gw-deprecatedExtensionsVersion adds "(Deprecated) " as a prefix to an operation's "summary" and to a JSON schema property's "title" (there is no equivalent for parameters).
  • Adding one or both of x-gw-deprecatedVersion and x-gw-deprecatedExtensionsVersion adds "(Deprecated as of x.y.z)" or similar suffix to the operation, parameter, or schema property's "description".
  • If x-gw-deprecatedReason is set, it is appended to the description suffix, such as "(Deprecated as of x.y.z - No longer supported)".
  • It is legal to set both x-gw-deprecatedVersion and x-gw-deprecatedExtensionsVersion, but it will generate a validation warning, since that's likely a configuration error.
  • It is not legal to set x-gw-deprecatedReason without also setting one of x-gw-deprecatedVersion or x-gw-deprecatedExtensionsVersion, except in the case of a Swagger operation that sets "deprecated" to true.
  • In Swagger operations, setting either x-gw-deprecatedVersion or x-gw-deprecatedExtensionsVersion implicitly sets the "deprecated" boolean to "true".
  • x-gw-deprecatedReason can be localized similar to other schema properties, using the property key suffix "deprecatedReason" (as "x-gw-deprecatedReason") would not be a legal part of a property key.
  • If both x-gw-sinceVersion and x-gw-deprecatedVersion (or their extensions equivalents) are present, both are appended to the description.

IA‑1852 By default, the REST API Framework rejects blank strings as input values for JSON string properties within REST API requests. The default is still to disallow blank strings, but the behavior can now be controlled through some explicit schema properties.

The new changes work as follows:

  • The "x-gw-allowBlankString" property can be set explicitly to a Boolean true/false value on a JSON Schema property or items.
  • The "x-gw-defaultPropertyAllowBlankString" property can be set to a Boolean true/false value at the root of the JSON Schema or Swagger schema. If set on a Swagger schema, it applies to schema definitions defined on that Swagger schema, but not to imported schemas. For array properties, whether blank items are allowed is only determined by the value of the "x-gw-allowBlankString" property. If not specified, the default is to allow blank strings for items.
  • For string properties, if "x-gw-allowBlankString" is specified explicitly, the property will use that value. The following describes how default values are determined when a value is not specified:
  • If the property specifies "enum" or "pattern", the default is to allow blank strings, on the assumption that the enum values or pattern already constrain the set of valid values and will generally disallow empty or blank strings.
  • Otherwise, if the containing schema specifies "x-gw-defaultPropertyAllowBlankString", that value is used as the default value for properties. Note that the default value can be overridden in a combining schema, so the default value for a property depends on the final value for the combined schema that property is in, which could be different from the default value of the schema where the property is actually defined.
  • If the containing schema does not specify that property, the default value is considered to be "false" and blank values are forbidden.

The overall effect of these changes is that an insurer can now specify "x-gw-defaultPropertyAllowBlankString": true at the root of their custom schema to allow blank strings by default for all properties in that schema, or they can enable or disable the blank string validation on a per-property basis.

IA‑1863 Fixed an issue with recursive schemas where the processing of its properties led to an infinite loop. Schemas are now tracked so that recursive processing is limited to a maximum of 20 levels of iteration.
IA‑1872 To prevent calls from sorting large numbers of rows, each call executes a limited COUNT(*) query prior to issuing the main query if the request is hitting a potentially large table and attempting to sort on columns that do not match a backing index. If the query returns too many results, the request will be rejected with an error message telling the caller to add additional restrictions to the query and/or to remove the sorts.

IA‑1875 Requests for included resources need to verify that the caller has permission to view the included thing. For the case of an inclusion coming from a schema property, this means both validating that the user can see the schema property and that the user can see the resulting included resource.

However, in the case of a nested inclusion like "include=states&include=states:governor", the check that the user could see the "governor" field was incorrect.
