Bulk Invoice Submission work queue

Code

BulkInvoiceSubmission

Categories

UIRunnable

Implementation

Work queue

Schedule

Not schedulable

Class

BulkInvoiceWorkQueue.java

The Bulk Invoice Submission work queue processes bulk invoice items for bulk invoice submission.

Processing each item consists of creating the placeholder check on that item's associated claim. After the approval of a bulk invoice, the Bulk Invoice Submission work queue runs against the bulk invoice and creates a work item for each bulk invoice item. Then, the work queue workers pick up each item and process that work item's bulk invoice item.

Note: If a bulk invoice remains in the PendingItemValidation status and all workers have finished, run the Bulk Invoice Submission work queue again.

ISCC-980 additions

The Bulk Invoice Submission work queue creates one work item for each BulkInvoiceItem on the BulkInvoice when a bulk invoice has one of the following statuses:

  • Initiating check creation
  • Initiating escalation
  • Initiating void
  • Initiating stop
  • Initiating delete

Because this is an asynchronous operation, these statuses are known as Asynchronous Processing.

The Bulk Invoice Submission work queue processes invoice items when a bulk invoice has one of the following statuses:
  • Creating checks
  • Processing escalation
  • Processing void
  • Processing stop
  • Processing delete

These statuses are known as Parallel Processing.

The BulkInvoiceStatus.tti typelist contains two typefilters (ParallellProcessing and AsyncProcessing) to identify these two sets of bulk invoice statuses. The work queue uses these typefilters to identify processing operations. Asynchronous Processing occurs first, followed by Parallel Processing. Parallel Processing begins when all work items are created for that BulkInvoice, and the Bulk Invoice Submission work queue is processing the work items and performing the particular operation on each BulkInvoiceItem.

When processing is complete, the BulkInvoice status will transition to the next status as shown in the following:
Parallel Processing Status Resulting Status after Processing
Processing delete Retired
Creating checks Awaiting Submission
Processing escalation Requesting
Processing void Pending void
Processing stop Pending stop
Note: To communicate downstream operations, the resulting 'Requesting', 'Pending void', and 'Pending stop' statuses can be used to configure the Messaging plugin. The Messaging plugin can be configured to call the appropriate acknowledge() method to update these statuses to 'Requested', 'Voided', and 'Stopped' (similar to the same statuses on Check). This configuration in both Message creation in Event Fired rules and the Messaging plugin would need be performed during customer implementation and is not available in the base configuration.
