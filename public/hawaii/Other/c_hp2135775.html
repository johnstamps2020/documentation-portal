User session replication

Do not attempt to replicate sessions across GenericCenter cluster members. GenericCenter does not implement or support user session replication for a number of reasons, including the following:

  • GenericCenter sessions are not serializable. Therefore, you cannot replicate a GenericCenter session, either with or without persistence to the database.
  • GenericCenter sessions hold the user state in memory and contain large amounts of information. Guidewire estimates that this information amounts to 1 MB of data on average for a 32-bit server and close to 2 MB for a 64-bit server. Session replication would create significant cross-member communication that is detrimental to performance.
  • GenericCenter commits changes to the database on almost all transactions. Notable exceptions are some wizards for which GenericCenter commits data changes only after the user completes all necessary entries.
  • GenericCenter scales horizontally almost linearly. The implementation of a session replication solution would very likely impede that linear scalability.

Instead, Guidewire recommends that you implement a GenericCenter cluster consisting of multiple GenericCenter application instances with failover and a load balancing solution. The load balancer must implement session affinity, meaning that it must route connections from the same user session to the same GenericCenter server.
