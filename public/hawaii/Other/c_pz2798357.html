Producer payment batch process

Performance was improved by performing commission work incrementally whenever the batch process is run rather than all at one time when generating a producer statement. The following sections describe application behavior and data changes resulting from the improvement.

Tracking earned and payable commission

Commissions on a policy period are earned and made payable to the producer at the time the commission is earned, rather than when the producer statement is created. As a result, BillingCenter still tracks payable commission at the producer level, but no longer tracks payable commission for a policy period or particular charge or invoice item.

A side effect of not tracking payable commission for a policy period is that it is no longer possible to request a write-off commission recovery for a policy period. Instead, the write-off recovery can be requested for a producer code, the process of which remains unchanged from earlier releases.

Automatic commission transactions

A commission transaction that generates earned commission at the ProducerCode, PolicyCommission, or ChargeCommission level triggers the automatic execution of a second transaction. The transaction moves the earned commission in the Commissions Payable T-account of the ProducerCode, PolicyCommission, or ChargeCommission to the Commissions Payable T-account at the Producer level. This operation results in the Commissions Payable T-accounts owned by ProducerCode, PolicyCommission, and ChargeCommission to consistently maintain zero balances. A new database consistency check is performed to verify that the zero balance is maintained. Also, all earned commission is maintained at the Producer level only.

The triggering transactions and the subsequent automatic transaction are listed in the following table. Reversing the triggering transaction also reverses the related automatic transaction.

Triggering commission transaction

Automatic transaction

ReserveCmsnEarned and

PolicyIncentivesEarned

A non-reversible PolicyCmsnPayable transaction.

ManualCmsnEarned (deprecated) and

ProdCodeBonusEarned

A non-reversible ProdCodeCmsnPayable transaction. This operation remains unchanged from earlier releases. For information about the deprecated ManualCmsnEarned transaction, see the ManualCmsnPayment description in Producer payment batch process.

PolicyCmsnPayable Transaction events

A PolicyCmsnPayable transaction no longer fires an ItemEventType of cmsnPayment, commissionMovedTo, or commissionMovedFrom. In addition, the ItemEventType typecode cmsnPayment has been retired.

ProducerStatement Related events and preupdate rules

In earlier BillingCenter releases, the Producer.Statements array contained only final producer statements. In BillingCenter 9.0, the Statements array contains both final producer statements and in-progress “pending” statements. Pending statements are differentiated from final producer statements by the setting of the ProducerStatement.Processed field. The field value for a pending statement is false, while a final producer statement has a field value of true.

Customization code related to the Statements array, such as handling code for a ProducerStatementAdded event, must be able to process or ignore each Statement array element, as appropriate.

Also, earlier releases added elements to the Statements array only when the final producer statement was generated by the Producer Payment batch process. In BillingCenter 9.0, elements can be added to the array outside of the batch process. Customization code that contains assumptions concerning the context that exists when an element is added to the array must be examined to verify its continued accurate execution.

Finally, a pending statement can ultimately be skipped and removed from the database. This characteristic of pending statements places certain limitations on the operations that can be safely performed on a pending statement’s ProducerStatement object. For example, extension fields added to the ProducerStatement must be set only on processed, non-pending statements. Related to these limitations, the ProducerStatement class extends the Validatable class which causes a preupdate rule to run whenever an instance of ProducerStatement is created or modified. The operations performed by any existing ProducerStatement preupdate rule must be examined to verify the operation’s adherence to the ProducerStatement limitations.

For details about the limited operations that can be performed on a pending statement, see ProducerStatement.

For details about preupdate rules, see Preupdate.

Entity, class, and field changes

The following entities and Gosu classes and fields have changed.

Changed entity, class, or field

Description

CommissionEarned entity

The entity has the following subtypes: ManualCmsnEarned (deprecated), PolicyIncentivesEarned, ProdCodeBonusEarned, ProducerAdvanceExpired, ProducerAdvancePayable, ProducerAdvanceRepaid, ProducerPayableTransferTxn, and ReserveCmsnEarned.

CommissionPayable entity

The entity has the following subtypes: PolicyCmsnPayable and ProdCodeCmsnPayable.

The ProducerPaymentSent edge foreign key and the associated edge table bc_pybltnxpmntsent have been removed. No version of BillingCenter since 3.0, inclusive, has persisted data to this table.

InvoiceItem and SnapshotInvoiceItem classes

Both classes have replaced the fields listed below.

  • PrimaryCmsnPayableAmount replaced with PrimaryAgencyBillRetained
  • PrimaryPaidCommission replaced with PrimaryDirectBillEarned

ItemCommission class

The class has replaced the fields listed below.

  • CommissionPaid replaced with AgencyBillRetained. The field value is computed from ChargePaidFromProducer transactions.
  • CommissionPayable replaced with DirectBillEarned. The field is computed from ReserveCmsnEarned transactions.

ManualCmsnPayment entity/class

The entity/class and the ManualCmsnEarned transaction it issues are deprecated for security reasons. They enabled a bonus-like payment to be issued without requiring approval. Instead, use the BonusCmsnPayment class.

Producer.SuspendNegativeAmounts field

Replaced with the HoldStatement field. Related to this field are the new fields StatementHoldPositiveLimit and StatementHoldNegativeLimit. After performing the upgrade, the special UpgradeProducerStatementHoldLimits batch process can be run a single time to initialize these three new fields. For details, see Producer payment batch process.

ProducerPaymentRecurrable class

This class is removed. The following fields from the class have been moved to the Producer class under different names. The moved ProducerPaymentRecurrable fields and the renamed Producer fields are listed below.

  • InitialDate renamed as InitialRecurDate
  • DayOfMonth renamed as RecurDayOfMonth
  • FinalDate renamed as FinalRecurDate
  • Periodicity renamed as RecurPeriodicity

ProducerStatement entity

The entity has been made Editable. It is no longer Retireable.

ProducerStatement.Amount field

Deprecated. This single field was performing multiple operations. It also introduced performance issues. Replaced with two new ProducerStatement fields. The value in either field is valid only on processed statements (ProducerStatement.Processed=true).

  • NetAmountSentAndReceived – Total net payments sent and received within the time period covered by the statement.
  • Balance – Remaining unpaid amount. The field can be non-zero if the net payment is negative or does not meet the minimum threshold for sending the payment.

UpgradeProducerStatementHoldLimits Batch process

The Producer class supports three new fields: HoldStatement, StatementHoldPositiveLimit, and StatementHoldNegativeLimit. During the upgrade operation, the HoldStatement field is assigned the boolean value of the field it replaces, SuspendNegativeAmounts. The other two fields are initialized to a monetary value of zero in the producer’s currency.

The initial values of the three new fields can be further modified by optionally running the UpgradeProducerStatementHoldLimits batch process a single time after the upgrade operation. The batch process is implemented in the Gosu class UpgradeProducerStatementHoldLimits, which can be customized. In the base configuration, the batch process assigns a monetary value of -100 in the producer’s currency to the StatementHoldNegativeLimit field for each Producer object whose HoldStatement field equals true.

See also Producer Payment.
