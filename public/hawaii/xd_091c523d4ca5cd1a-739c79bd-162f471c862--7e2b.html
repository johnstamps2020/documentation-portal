New and changed in integration in 10.0.0

New in integration in 10.0.0

Guidewire InsurancePlatform Integration Views

Integration Views allows you to define a stable, versioned contract for publishing data to and retrieving data from the GenericCenter database. In addition, the feature provides a simple, declarative way to map underlying GenericCenter data into a serialized format that conforms to the contract.

The technical realization of this contract starts with two mandatory components and an optional component. The mandatory components include a mapping and a schema. The optional component is a filter. The mapping is a set of declarations that transform an object of a particular type first into an intermediate object that conforms to a specified schema. In the Integration Views context, the ultimate serialized form of this intermediate object is a JSON object or XML element. The schema defines the structure of the data involved in the custom processes. The schema also enables you to use off-the-shelf tools to generate objects and methods to work with the data once it is serialized. The optional filter specifies the parts of the schema to use as output or for presentation. The end result of an Integration View is a JSON object or XML element.

Marshal and unmarshal Gosu objects to and from XML

The marshal and unmarshal methods of the WsiExportableUtil class can convert additional types of Gosu objects to and from XML.

The methods could always process Gosu DTOs (data transfer objects). The following additional types can also be processed.

  • Simple types, such as int, Integer, Double, String, boolean, and Boolean
  • GWRemotable types
  • GX models
  • XSD types
  • Lists of marshallable types

Support for testing JMS inbound integration in non-production mode

GenericCenter now supports running inbound integration in dev mode on the Jetty server that the base configuration of Guidewire Studio provides. You can use this environment to test your implementation of inbound integration while you develop your code. Use one of the unmanaged thread pool types, not a J2EE managed thread pool.

You cannot use the Jetty server in production mode. You must use one of the supported application servers: JBoss, WebLogic, or WebSphere.

Additional directory in plugins/Gosu for internal use only

The GenericCenter/modules/configuration/plugins/Gosu directory now has a subdirectory called gresources. This new subdirectory is for internal use as a compilation output path. Never use, modify, or rely upon the contents of this directory.

Changed in integration in 10.0.0

Changes to servlet definitions in servlets.xml

The configuration/config/servlet/servlets.xml file now supports env and server attributes and static parameters that GenericCenter uses to configure a servlet at start up.

The env attribute is an optional comma-separated list of names of environments for which the servlet definition applies. The server attribute optionally specifies the server for which the servlet definition applies. The value of this attribute is either a server role preceded by the # character or a server ID.

The servlets.xml file in the base configuration contains an example of the use of <param> subelements in the definition of the InfoServlet servlet.

GenericCenter now supports running JMS inbound integration on a JBoss application server. Use the <j2ee-managed-threadpool> element to define the thread pool for this integration.

Changes to BillingCenter web services

To obtain previews of payment plans from BillingCenter, GenericCenter uses the BillingAPI web service to send account information to BillingCenter. The owner account or the alternate billing account may not exist in BillingCenter. New with this release, if the account does not exist, GenericCenter sends account information to BillingCenter so BillingCenter can use the account information to generate an accurate preview of the payment schedule. In BillingCenter 10.0, the BillingAPI methods to preview renewal, rewrite, and issuance invoices now include optional ownerAccount and overridingPayer parameters for the owner and alternate billing accounts if they do not exist in BillingCenter. The BillingAPI methods are:
  • previewIssuanceInvoices
  • previewRenewalInvoices
  • previewRewriteInvoices

In the ownerAccount and overridingPayer parameters, GenericCenter passes the billing level (BillingLevel) to BillingCenter. From then on, BillingCenter manages the billing level, and GenericCenter retrieves it from BillingCenter. BillingLevel is included in the BillingAPI methods to preview invoices and createAccount. BillingLevel is also included in the retrieveAccountBillingSummary method in the BillingSummaryAPI.

See the BillingCenter documentation for more information on the changes to the BillingAPI.

Changes to PAS integration

The PCNewAccountInfo parameter

As of this release, for all new accounts created through the BillingAPI, the billing level is set by the policy administration system (PAS). The BillingAPI.createAccount method now receives a new type of parameter called PCNewAccountInfo, which includes a required BillingLevel option.

Modifications to the invoice preview methods

The policy administration system may want to see what invoices would look like for a new policy period before binding and issuing the policy transaction. BillingCenter provides a set of invoice preview methods to support this. They are: previewIssuanceInvoices, previewRenewalInvoices , and previewRewriteInvoices. Each of these methods generates a set of preview invoices based on the settings of the policy period's intended owner and payer accounts.

In some situations, the policy administration system may want to preview invoices for an owner account and/or a payer account that does not yet exist in BillingCenter. As of this release, the invoice preview methods have been modified to receive two optional PCNewAccountInfo objects:

  • ownerAccount: This PCNewAccountInfo object represents the owner account. This parameter can be used when the policy period owner does not exist in BillingCenter. This parameter lets you provide owner account settings that may impact how invoices are generated.
  • overridingPayer: This PCNewAccountInfo object represents the overriding payer account. This parameter can be used when the overriding payer does not exist in BillingCenter. This parameter lets you provide overriding payer settings that may impact how invoices are generated.

Both parameters are optional. Be aware that if the accounts do not exist in BillingCenter and you do not provide values for these parameters, BillingCenter generates invoice previews based on default settings that BillingCenter uses when creating new accounts. For example, when a new account is created through the BillingAPI, the default setting for the BillingLevel field is policy-level billing. When using the preview methods, if no parameter is specified for the owner account, the invoice preview methods assume the owner account uses policy-level billing.

To generate the most accurate invoice previews, Guidewire recommends setting these parameters whenever the owner or payer accounts are not yet known to BillingCenter.

Changes to holding charges for final audits

For policy transactions that modify existing policy periods, there may be circumstances where the policy administration system does not want to schedule the invoice items as normal. In these situations, the billing instruction includes a special handling instruction. This instruction overrides the way in which invoice items are scheduled. For more information on special handling instructions, refer to the Application Guide.

When a policy requires a final audit, BillingCenter recommends that the billing instruction include a special handling instruction to Hold for Final Audit (All Unbilled Items). This instruction tells BillingCenter to place charge holds on all charges on this billing instruction and all unbilled invoice items on the policy period. These holds remain until the final audit is received. Additional activity is halted prior to the final audit so that there is only one final invoice or refund and this invoice or refund has the correct amount.

In previous releases, the CancelInfo DTO has a HasScheduledFinalAudit Boolean attribute. Setting this attribute to true places a Hold for Final Audit (All Unbilled Items) hold on the policy's charges.

As of this release, the HasScheduledFinalAudit attribute has been replaced by a SpecialHandling typekey attribute. To place a Hold for Final Audit (All Unbilled Items) hold on the policy's charges, set the SpecialHandling attribute to HOLDFORAUDITALL.

Be aware that, in previous releases, the following DTOs also have a HasScheduledFinalAudit attribute: PolicyChangeInfo, FinalAuditInfo, ReinstatementInfo, PremiumReportInfo, PolicyPeriodGeneralInfo. A Hold for Final Audit (All Unbilled Items) hold makes business sense for cancellations only. Therefore, in previous releases, setting this attribute to true for these types of DTOs has no effect.

As of this release, the HasScheduledFinalAudit attribute has been replaced by a SpecialHandling attribute for all of these DTOs. For PolicyChangeInfo, FinalAuditInfo, ReinstatementInfo, PremiumReportInfo, and PolicyPeriodGeneralInfo, setting the SpecialHandling to HOLDFORAUDITALL has no effect.

In other words, in both previous releases and the current release, you have the ability to specify a Hold for Final Audit (All Unbilled Items) hold for both cancellation and non-cancellation DTOs. But, BillingCenter creates a Hold for Final Audit (All Unbilled Items) hold for cancellations only, even when a hold is specified in a non-cancellation DTO.

Changes to billing system integration

Several changes were made to billing system integration.

New typelists

  • InvoicingMethod – Whether to use default billing, custom billing, or override the invoice stream
  • BillingLevel – How to bill policies on the account
  • BillDateOrDueDateBilling – Corresponds to typelist in BillingCenter

Changes to StandAloneBillingSystemPlugin

Modified and added more payment plans to the StandAloneBillingSystemPlugin plugin:
  • Twice Per Month Demo – Statement dates are now reliably twice per month. By using Invoicing Overrides, you can set the statement dates. Previously, the statement dates were determined by dividing the range in days by the number of payments, and sometimes missed the expected payment dates.
  • Quarterly Demo – New. Send statements quarterly.
  • Monthly Demo – New. Send statements monthly.
  • Every Other Week Demo – New. Send statements every other week.
