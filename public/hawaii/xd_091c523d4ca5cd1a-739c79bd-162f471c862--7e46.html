Changed in the application in 10.0.0

Changes to high volume quote requests

This release includes the following changes for handling high volume quote requests:

  • User interface for searching the quote store
  • Additional methods in the API

Expanded BillingCenter integration

GenericCenter integration with BillingCenter now provides improved control over the invoice stream and payment instrument.

From GenericCenter, you can:
  • With policy-level billing, you can override defaults for the invoice stream of each policy. These overrides are propagated to BillingCenter.
  • Specify how to invoice excess premium in a policy change.
  • Capture the payment instrument.

Multicurrency integrations now provide a more consistent experience in creating organizations and producer codes.

Multicurrency changes

BillingCenter now enables agency bill and commission plans to be defined in multiple currencies. GenericCenter makes use of these improvements and can associate multicurrency plans with producers and organizations. In previous releases, you had to define a plan for each currency.
If a plan offers more than one currency, you can now select multiple currencies on these screens:
  • Organization > Agency Bill Plans
  • Producer Code > Commission Plans

The plugins that communicate with BillingCenter have been updated to handle these changes.

Payment screen changes

The Payment screen of a policy transaction has changed:
  • New read-only Billing Level on the Billing summary indicates whether to bill the policy separately or on an account bill. The IBillingSystemPlugin plugin implementation retrieves the billing level from the billing system. If this is a new policy, the plugin sets a default value. When integrated with BillingCenter, for existing policies the billing level comes from BillingCenter.
  • New Override Change in Cost Billing appears in policy change, issuance, and reinstatement policy transactions. The Special Handling field enables you to override how to bill the changes in cost. The overrides are propagated to BillingCenter. The default billing splits the cost change proportionally among the payments. Bill Immediately creates a new payment for the whole cost change amount. Bill on Next Invoice adds the payment to the next scheduled invoice.
  • New Hold for Final Audit billing override option appears for policies that have the schedule final audit set to Yes.
  • Invoicing Overrides replaces Custom Billing Options. You can override the default invoicing by specifying whether to send invoices on a specified date or relative to the due date. You can also override the payment instrument.
  • New Fix Invoices by replaces Due Date in the Invoicing summary. The value comes from the BillDateOrDueDateBilling typelist on the payment plan (PaymentPlanSummary).
  • In the Payment Schedule listing, you can only view the Schedule for the selected Payment Plan.
  • If you specify an Alternate Billing Account, the policy gets the Billing Level from the alternate account.

Restoring Custom Billing Options

In this release, Custom Billing Options on the Payment screen has been deprecated and replaced by Invoicing Overrides. If you used Custom Billing Options in a previous release and wish to restore Custom Billing Options and the related functionality, set the EnableCustomBilling parameter to true in config.xml.

In BillingCenter, this requires implementation beyond what is available in the base integration. Refer to KB 000006544 "BC8 Config Series: Policy Level Cash Separation and Custom Billing" for information on implementing custom billing in BillingCenter.

Inference tab for form patterns changes

In this release, form inference now includes inferring forms by:

  • Policy type or coverage part
  • Additional insured and additional interest
  • Based on whether a selected coverage, condition, or exclusion is not used on the policy

See also

  • Inference tab for form patterns

Generic schedule changes

This release includes generic schedules that you can work with in Product Designer. In Product Designer, you can add new scheduled clauses to a line of business. Scheduled clauses are clauses with a generic schedule attached. You can adapt the generic structure to represent many different schedules. In the base configuration, the Homeowners line of business provides examples of scheduled clauses that use generic schedules.

See also

  • Generic schedules

Improved labeling

Based on input from insurers and implementation teams, Guidewire has modified a number of labels in the user interface. The labels were modified to achieve multiple goals:

  • To move towards a language that is more business friendly
  • To use terminology consistently across different screens
  • To clarify the business data that is shown and the actions one can take

Changes are made only to display keys and a small number of typecode names and descriptions. No entity field names or typecode codes are modified.

The following section provides a high-level summary of these changes.

Summary of labeling changes

Invoice item "event dates" are now "placement dates"

The InvoiceItem entity has an EventDate field. This field is used to determine which invoice to assign the invoice item to.

In previous releases, this field is labeled Event Date in the user interface. As of this release, the field is labeled Placement Date to make it clearer to business users what the purpose of the field is.

Billing levels and cash separation

In previous releases, the Account Details screen identifies the billing level of an account and its use of cash separation in the following fields and settings:

  • Invoice By
    • When set to Account, the account uses account-level billing.
    • When set to Policy/Group, the account uses policy-level billing.
  • Separate Incoming Funds By
    • When set to Account, cash separation is disabled.
    • When set to Policy/Group, cash separation is enabled.

As of this release, these labels have been modified as listed below to directly name the settings.

  • The Invoice By label is now Billing Level
    • The Account radio button is now Account-Level Billing.
    • The Policy/Group radio button is now Policy-Level Billing.
  • The Separate Incoming Funds By label is now Cash Separation
    • The Account radio button is now Disabled.
    • The Policy/Group radio button is now Enabled.

"Context" is now "billing instruction"

There are several screens that display information based on different billing instruction types (such as issuance, policy change, cancellation, and so on). This includes:

  • The Charges screen, where you can see the billing instruction type for each charge
  • The Invoices screen, where you can see the billing instruction type for each invoice item.
  • The Payment Plan screen, where you can specify overrides for different billing instruction types.

In previous releases, the billing instruction type is labeled Context. As of this release, it is labeled Billing Instruction.

Improvements to Payment Plan screen

A number of changes were made to the Payment Plan screen, including the following:

  • There is now a Final Installment label to identify the fields that determine when the final installment must be billed or due.
  • The installment fee section is now labeled Installment Fees. (In previous releases, it is labeled Invoicing Fees. This can cause confusion with invoice fees, which are a different type of fee.)

Improvements to Billing Plan screen

A number of changes were made to the Billing Plan screen, including:

  • The automatic disbursements section, which was labeled Disbursements, is now labeled Automatic Disbursements. This was done to clarify that the settings affect automatic disbursements only.
  • The labels in the Automatic Disbursements section have been reordered so that similar fields appear together. Also, the labels have been reworded to make them shorter. For example:
    • The Calculate amount available for disbursement as label is now Amount Available.
    • The Automatic disbursement when surplus exceeds amount available for disbursement plus label is now Creation Threshold.
  • In the Payment Request Status section:
    • Two static labels have been added to remind users when a payment request's status is set to Created and Closed.
    • The other labels have been renamed to clarify they are offsets from the Created date or the Closed date.

PCF files whose mode calls getPCFMode

As of this release, BillingCenter includes a small number of input sets and button sets that contain the following value for the mode attribute:

gw.plugin.Plugins.get(gw.module.IFeesThresholds).getPCFMode()

This includes input sets and button sets in the following PCF files (which display values related to billing plans or delinquency plans):

  • BillingPlanDetailScreen.pcf
  • BillingPlanDetailDV.pcf
  • AccountDetailInvoices.pcf
  • DisbursementDetailDV.pcf
  • DelinquencyPlanDetailScreen.pcf
  • DelinquencyPlanDetailDV.pcf
  • DelinquencyProcessDetailsDV.pcf

These input sets and button sets do not have any special behavior in the current release. They have been added to support "Feature Modules", an upcoming capability that may be added to BillingCenter in the future. Guidewire recommends that you leave them place and that you not modify the mode attribute. If for some reason you do need to modify one or more of these sets in this way, Guidewire recommends that you discuss the impact with your Guidewire representative.

Future development support and upgrading insurers

If an upgrading insurer has used any of the following methods in their own configurations, the code will need to be refactored. For details on the new methods to use, see The IFeesThresholdsPlugin.

  • The IInvoice plugin's getInvoiceFeeAmount method
    • All usage of the method must be refactored.
  • The IInvoice plugin's getLateFeeCharge method
    • Code that retrieves the late fee charge amount must be refactored.
    • Code that retrieves only the late fee charge's charge pattern, owner, or payer does not need to be refactored.

The IAccount plugin has been removed

In some previous release, the IAccount plugin is deprecated. As of this release, it has been removed.

  • The getDistributionLimitType method is now declared in the AccountEnhancement class as DistributionLimitTypeFromPlan.
  • The shouldHold(account) and shouldHold(unappliedFund) methods have been moved to the IAutomaticDisbursement plugin.

Refactoring of builder methods

As of this release, several changes have been made to the data builders provided in the base configuration. These changes have been made to facilitate and standardize the way they are used. Several builders have been deprecated, with alternatives being documented in each of the deprecation messages. In addition to that, some of the builders were removed, with documentation about replacements being provided in the upgrade diff report.

Changes to PAS integration

The PCNewAccountInfo parameter

As of this release, for all new accounts created through the BillingAPI, the billing level is set by the policy administration system (PAS). The BillingAPI.createAccount method now receives a new type of parameter called PCNewAccountInfo, which includes a required BillingLevel option.

Modifications to the invoice preview methods

The policy administration system may want to see what invoices would look like for a new policy period before binding and issuing the policy transaction. BillingCenter provides a set of invoice preview methods to support this. They are: previewIssuanceInvoices, previewRenewalInvoices , and previewRewriteInvoices. Each of these methods generates a set of preview invoices based on the settings of the policy period's intended owner and payer accounts.

In some situations, the policy administration system may want to preview invoices for an owner account and/or a payer account that does not yet exist in BillingCenter. As of this release, the invoice preview methods have been modified to receive two optional PCNewAccountInfo objects:

  • ownerAccount: This PCNewAccountInfo object represents the owner account. This parameter can be used when the policy period owner does not exist in BillingCenter. This parameter lets you provide owner account settings that may impact how invoices are generated.
  • overridingPayer: This PCNewAccountInfo object represents the overriding payer account. This parameter can be used when the overriding payer does not exist in BillingCenter. This parameter lets you provide overriding payer settings that may impact how invoices are generated.

Both parameters are optional. Be aware that if the accounts do not exist in BillingCenter and you do not provide values for these parameters, BillingCenter generates invoice previews based on default settings that BillingCenter uses when creating new accounts. For example, when a new account is created through the BillingAPI, the default setting for the BillingLevel field is policy-level billing. When using the preview methods, if no parameter is specified for the owner account, the invoice preview methods assume the owner account uses policy-level billing.

To generate the most accurate invoice previews, Guidewire recommends setting these parameters whenever the owner or payer accounts are not yet known to BillingCenter.

Changes to holding charges for final audits

For policy transactions that modify existing policy periods, there may be circumstances where the policy administration system does not want to schedule the invoice items as normal. In these situations, the billing instruction includes a special handling instruction. This instruction overrides the way in which invoice items are scheduled. For more information on special handling instructions, refer to the Application Guide.

When a policy requires a final audit, BillingCenter recommends that the billing instruction include a special handling instruction to Hold for Final Audit (All Unbilled Items). This instruction tells BillingCenter to place charge holds on all charges on this billing instruction and all unbilled invoice items on the policy period. These holds remain until the final audit is received. Additional activity is halted prior to the final audit so that there is only one final invoice or refund and this invoice or refund has the correct amount.

In previous releases, the CancelInfo DTO has a HasScheduledFinalAudit Boolean attribute. Setting this attribute to true places a Hold for Final Audit (All Unbilled Items) hold on the policy's charges.

As of this release, the HasScheduledFinalAudit attribute has been replaced by a SpecialHandling typekey attribute. To place a Hold for Final Audit (All Unbilled Items) hold on the policy's charges, set the SpecialHandling attribute to HOLDFORAUDITALL.

Be aware that, in previous releases, the following DTOs also have a HasScheduledFinalAudit attribute: PolicyChangeInfo, FinalAuditInfo, ReinstatementInfo, PremiumReportInfo, PolicyPeriodGeneralInfo. A Hold for Final Audit (All Unbilled Items) hold makes business sense for cancellations only. Therefore, in previous releases, setting this attribute to true for these types of DTOs has no effect.

As of this release, the HasScheduledFinalAudit attribute has been replaced by a SpecialHandling attribute for all of these DTOs. For PolicyChangeInfo, FinalAuditInfo, ReinstatementInfo, PremiumReportInfo, and PolicyPeriodGeneralInfo, setting the SpecialHandling to HOLDFORAUDITALL has no effect.

In other words, in both previous releases and the current release, you have the ability to specify a Hold for Final Audit (All Unbilled Items) hold for both cancellation and non-cancellation DTOs. But, BillingCenter creates a Hold for Final Audit (All Unbilled Items) hold for cancellations only, even when a hold is specified in a non-cancellation DTO.

Changes to Bulk Invoice processing

The processing of bulk invoices has been enhanced and is now managed asynchronously with the introduction of the InvoiceItemWorkQueue work queue.

The InvoiceItemWorkQueue work queue consumes InvoiceItemWorkItem entities and now supports the following additional processes:
  • Check creation
  • Voiding of checks
  • Stopping of checks
  • Financial escalation
  • Deletion of checks

Changes to GenericCenter business rules

In GenericCenter versions prior to 10.0.0, the rule list screen showed an Enabled column. Guidewire has removed this column in GenericCenter 10.0.0. Instead, in 10.0.0, the Run Status column icon (and tooltip) indicate whether a rule is enabled or disabled.

Also new in 10.0.0 is the ability to directly enable or disable a business rule from the rule list screen or the rule details screen:
  • From the rule list screen, by selecting More > Enable Selected or More > Disable Selected.
  • From the rule detail screen, by placing the rule in edit mode and updating the Enabled check box.

Clicking the Enable / Disable button on a rule detail screen toggles the Enabled field on the selected version of the rule. The label of the button changes depending on whether the selected rule version is currently enabled or disabled.

Business rules and rolling upgrade

In 10.0.0, during a rolling upgrade, Guidewire disables the ability to deploy a business rule in a production environment. The intent is to prevent the deployment of a rule that contains data that is incompatible with the current environment, thereby causing the rule execution to fail.

During a rolling upgrade, GenericCenter:
  • Disables the Deploy All and Deploy Selected options on the rule list screen.
  • Disables the Deploy button on the rule detail screen.
  • Displays an alert message on the rule list and rule details screen that indicates rule deployment is not possible due to an ongoing upgrade.
